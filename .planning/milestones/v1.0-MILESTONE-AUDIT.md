## E2E Flows

### Flow 1: Normal download run

```
bcdl username
  → shutil.which("yt-dlp") check — exits code 1 if absent
  → args.format validation (skipped if no --format)
  → get_all_collection_items(username)
  → load_state(.bcdl/username.json)
  → item loop:
      if item_id in state → print "[skip]", increment skipped     [RESM-01]
      else → download_with_retry(item, i, total, audio_format)    [RELY-01, DCTL-01]
          → _run_yt_dlp(cmd) with retry and backoff
          → if success: save_state(state, path)                   [RESM-01]
  → print "Done: N downloaded, N skipped, N failed."
```

Status: COMPLETE

### Flow 2: Resume after interruption

```
Run 1: item downloads → state entry written atomically
Run 2: load_state reads same file → item already in state → "[skip]"
```

Status: COMPLETE. Failed items not written to state, so they are retried on next run.

### Flow 3: Format selection

```
bcdl --format flac username
  → args.format = "flac"
  → validated against SUPPORTED_FORMATS = ("flac", "mp3", "wav", "aac", "opus")
  → audio_format="flac" passed to download_with_retry
  → cmd += ["-x", "--audio-format", "flac"]
  → yt-dlp converts to FLAC
```

Status: COMPLETE

## Tech Debt (No Critical Blockers)

### Phase 02: `download_item` dead code

`bcdl.py:238-258` — `download_item` is unreachable in production. `main()` calls `download_with_retry` (line 340) exclusively. `TestDownloadItem` (test_bcdl.py:197-236) tests this dead function directly using the old `subprocess.run` pattern without retry, stdout suppression, or clean status output. These tests pass but provide false coverage signals.

**Risk:** A future developer seeing `TestDownloadItem` passing may assume the function is live. It is not.
**Suggested resolution:** Remove `download_item` and `TestDownloadItem`, or add a deprecation comment.

### Phase 03: `human_needed` verification status

`03-VERIFICATION.md` frontmatter shows `status: human_needed`. This is because the pipx smoke test requires interactive confirmation. The user approved it during execution. The verification document is not stale — it correctly reflects that automated verification cannot close this item. No action required unless independent confirmation is desired.
